## Context

`llm-providers` shipped a client that resolves to a `RoleClient`; nothing called `complete()` more than once. This change is the loop -- everything about how many turns, what happens on a tool error, when context gets pruned, and how a screenshot gets represented lives here.

## Decision 1: Tool execution goes directly against `engine::Session`, not through MCP's `#[tool]` wrappers

`crates/mcp`'s `#[tool]` methods on `AibTools` are private, take rmcp's `Parameters<Req>` wrapper, and return `CallToolResult` (MCP content blocks) -- none of which is meaningful to call from a plain Rust loop. `crates/agent/src/tools.rs` calls `engine::Session`'s own public methods directly (`navigate`, `click`, `type_text`, ...), the same methods the MCP wrappers themselves call. `mcp-task-delegation` (change 4) will thread the *same* `engine::Session` (via `SharedSession`) into both the harness and the existing `AibTools` machinery, so an outer MCP host and `aib`'s own driver can share one browser -- but that sharing is change 4's concern; this change only needs `SharedSession` to exist in the exact shape `AibTools` already stores its session in (`Arc<Mutex<Option<Session>>>`), confirmed against `crates/mcp/src/lib.rs` before writing `types.rs`.

## Decision 2: Recoverable engine errors are tool-result text, not `Err`

`execute_tool`'s design principle: only two things abort the whole run (`Err(AgentError)`) -- the session is gone, or the model called a tool name genuinely not in `tool_defs()`. Everything else a real action can fail at (a stale ref because the DOM changed, a `wait_for` timing out, a failed `assert`, an unknown page id) comes back as `Ok(ToolOutcome::Text("error: ..."))`. This is the entire reason to run a loop instead of a fixed script: the model sees the failure and can take a fresh snapshot, retry, or give up via `task_failed` -- deciding that is the model's job, not the harness's. Malformed tool-call argument JSON gets the same treatment (a corrective error text, not a crash) since a provider's function-calling output quality varies and isn't something this project controls.

## Decision 3: Vision routing reads `RoleClient.vision`, not a separate flag threaded through the call

The user confirmed this explicitly during planning: screenshot routing is governed by the *driver role's own* `vision` capability flag (already part of `RoleClient` since `llm-providers`), not a separate per-call decision. `driver.vision == true` -> the image goes inline (on a follow-up `user` message, not the `tool`-role result -- OpenAI-compatible providers don't reliably accept image parts on a `tool` message, a documented workaround, not a guess). `driver.vision == false` -> if a `vision` role is configured, the image is sent there and its text interpretation becomes the tool result; if not, a clear tool-result error explains how to fix it (`[roles.vision]`), rather than silently doing nothing useful with the screenshot.

## Decision 4: Context pruning is unconditional and runs every turn, not on a token-count heuristic

`prune_snapshots` elides every snapshot-shaped tool result older than the most recent `max_retained_snapshots`, checked at the top of every loop iteration -- not triggered by an estimated token count (which would need a tokenizer per provider, real complexity for marginal precision gain) or by a fixed turn interval (which either prunes too eagerly on a short task or not enough on a long one). Since snapshots are the dominant source of context growth in this loop (every other tool result is short: "ok", an error string, a short list), bounding *snapshot count* specifically, unconditionally, is a simple rule that scales with the actual problem regardless of task length.

## Decision 5: `--driver`/`--vision provider/model` needed a real `llm-providers` extension, not a workaround

The CLI's ad-hoc override (`aib agent --driver deepseek/deepseek-chat "task"`, no `[roles.driver]` required) needed a way to resolve a `RoleClient` from a provider name + a model string directly, bypassing `[roles.*]`. Rather than duplicating `resolve_role`'s provider-lookup-and-client-construction logic inside `crates/agent` or `src/agent_cmd.rs` (which would drift from `resolve_role`'s own behavior over time -- e.g. if a future change adds a new `ProviderKind`, only one of the two copies would get updated), `crates/llm/src/config.rs` was refactored: the shared "look up this provider, build its client" logic moved into one private `resolve_client_for_provider`, parameterized by an `Option<&str>` role name for error-message purposes (`Some("driver")` when going through `[roles.*]`, `None` for a direct provider lookup, each producing a differently-worded, still-clear error). `resolve_role` and the new `resolve_provider_model` are now both thin callers of the same logic. This is a `llm-providers` capability change, tracked as a MODIFIED capability in this change's spec delta rather than silently bundled in.

## Testing note

Verified against a real Chrome session and a real local HTTP LLM stub (`crates/agent/tests/support/llm_stub.rs`, new -- can't reuse `crates/engine`'s or `crates/llm`'s own test-only stub servers, since each is private to its own crate's `tests/` directory) for the full step loop, error feedback, context pruning (using distinctly-marked `data:` URL pages per navigation so an elided snapshot is actually distinguishable from a retained one -- a same-content fixture navigated repeatedly would make this untestable), termination (no-progress and max-steps), and both vision-routing paths (inline image for a vision-capable driver, text interpretation via a second stub acting as the vision role for a non-vision driver) -- eight integration tests total, all live-verified, not mocked at the `Harness` boundary. One real test bug was found and fixed while writing the pruning test: asserting a marker string was absent from an elided snapshot initially failed because that same string legitimately still appears elsewhere in the conversation (the assistant's own earlier tool-call *arguments*, i.e. the navigate URL itself) -- not a harness bug, a too-broad assertion; fixed by checking for the walker's specific rendered-snapshot line shape instead of the bare marker string. The full `aib agent` CLI was also manually smoke-tested end-to-end (real separate-process Python HTTP stub, real headless Chrome, both human-readable and `--json` output modes), confirming the complete config-load -> role-resolution -> browser-launch -> step-loop -> exit-code path works outside the automated suite too.
