# Design: Popup Auto-Attach

## Context

The first slice of Phase 5 (Hardening). Every prior phase built strictly on top of Phase 1's "one browser, one context, one page" `Session` (design.md Decision #1 explicitly deferred a daemon/multi-session model, but that decision was about *processes*, not about a single process tracking more than one page -- `Session`'s own API was written, per that same decision, so a future daemon could wrap multiple `Session`s; this change instead teaches one `Session` to track multiple *pages* within itself, which is a smaller, compatible step, not a reversal of Decision #1).

## Goals / Non-Goals

**Goals:** notice a new top-level target (popup/new tab) opened as a side effect of interacting with the current page; let an agent explicitly discover and switch to it; keep every existing single-page behavior and MCP tool signature unchanged when no popup ever opens.

**Non-Goals:** cross-origin OOPIF (iframe) attach (see proposal.md's scope reduction); automatic popup cleanup/GC; auto-switching to a newly-opened popup without the agent asking.

## Decisions

1. **Discovery is browser-wide and observe-only (`Target.setDiscoverTargets`), not `Target.setAutoAttach`.** `Target.setDiscoverTargets({discover: true})` is called once on the browser-level session (in `Browser::new_context`), and `Target.targetCreated`/`Target.targetDestroyed` events are filtered client-side to this session's own `browserContextId` and `target_type == "page"`. When a match is found, this client makes its own explicit `Target.attachToTarget` call and enables `Page`/`Runtime` domains itself -- the exact same path `new_page` already uses for its own primary page, just triggered by a discovery event instead of direct caller intent. **This replaces an original design of page-scoped `Target.setAutoAttach`, which live testing showed doesn't work the way assumed here -- see the Addendum below.**
2. **Discovery events are consumed lazily, not via a background task.** This project has no persistent event loop -- an MCP tool call is the only time the process does anything (`design.md` Decision #1's "no daemon" reasoning applies here too). Rather than adding one, `BrowserContext` exposes bounded event streams (the existing `events::<E>()` machinery already used for `Page`/`Network`/`Fetch` events) for `Target.targetCreated`/`Target.targetDestroyed`, subscribed once at context creation, and `Session` drains whatever has arrived so far, non-blockingly, at the start of `list_pages()`/`switch_page()`. An agent's natural workflow -- click something that might open a popup, then call `browser_list_pages` to check -- already provides the "ask again if it's not there yet" retry point; no artificial polling wait is added for v1.
3. **`Session` gains a page registry (`HashMap<target_id, Page>`) plus an `active_target_id`, both behind the same interior-mutability pattern already used for `mouse_pos`/`training_active` (`Mutex`), not `&mut self`.** Every existing action method (`click`/`type_text`/`snapshot`/`navigate`/etc.) resolves against "the active page" through a small `active_page()` helper instead of a bare `self.page` field access -- mechanical to apply, but touches most of `session.rs`. External behavior for the common single-page case (nothing ever calls `browser_switch_page`) is unchanged.
4. **A dedicated `primary_target_id` (the page `Session::launch` originally created) is tracked separately from `active_target_id`.** If the currently-active page detaches (the popup closes itself, e.g. after an OAuth redirect completes), `active_target_id` resets to `primary_target_id` rather than to an arbitrary remaining page -- a predictable fallback mirroring how closing a popup naturally returns a real user's focus to the tab that opened it.
5. **`browser_list_pages`/`browser_switch_page` use `page_id` as the external parameter name, mapped internally to CDP's `targetId`.** Matches this project's existing convention of translating CDP jargon into agent-friendly terms (e.g. `ref` instead of `backendNodeId`).
6. **No proactive staleness check on every action call for a switched-to page that has since closed.** `switch_page` validates the requested `page_id` exists in the registry at switch time (cheap, valuable at the point of intent); if a page closes *after* being switched to and an action is attempted anyway, the resulting CDP-level protocol error surfaces via the existing `EngineError::Cdp` variant rather than a bespoke pre-check on every call -- acceptable for v1, consistent with keeping this slice reduced.

## Risks / Trade-offs

- [The exact CDP session/event-routing behavior of page-scoped `setAutoAttach` + flattened `attachedToTarget` is based on documented/observed CDP behavior (matching Puppeteer's implementation), not yet exercised against this project's own hand-rolled `Connection`/session-demuxing code] → real risk of a live-testing correction, same as `yaml-runner`'s `serde_yaml` surprise and `true-user-input`'s three live-testing bugs -- will be verified against a real popup-opening fixture before this is considered done, and any correction will be documented in a design.md addendum, not silently patched.
- [`session.rs`'s `self.page` → `self.active_page()` conversion touches most of the file] → mechanical, low-risk-per-line, but large diff; mitigated by running the full existing test suite (which exercises every action method) after the refactor, before adding any new popup-specific behavior on top.

## Migration Plan

Purely additive at the MCP surface (two new tools); the `Session` struct's internal field shape changes, but its public method signatures don't, so no caller-visible break for the single-page case.

## Open Questions

None blocking -- CDP session-routing specifics will be confirmed empirically during implementation, per the Risks section above.

## Addendum: three real bugs found during live verification

As anticipated in the Risks section, live testing against a real popup-opening fixture disproved the original mechanism and surfaced two further races. All three are fixed in the shipped code; documented here per this project's convention of narrating real bugs rather than silently rewriting the design text.

1. **`Target.setAutoAttach` conflicts with this project's own explicit-attach flow for the primary page, and hangs.** The original Decision #1 assumed calling `setAutoAttach` on a page's own session (matching a documented Puppeteer implementation detail) would scope auto-attach to that page's own popups without side effects. Live testing showed the opposite: enabling `setAutoAttach` (tried both page-scoped and, when that produced zero events, browser-scoped as a follow-up experiment) caused CDP to *also* try to manage the primary page's own already-explicitly-attached session, and the very next `Input.dispatchMouseEvent` call on it hung until the 30s command timeout. Diagnosed with a throwaway example (`crates/cdp/examples/diag_popup.rs`, removed after use) that isolated `window.open()` + `Target.getTargets()` from the attach-event machinery entirely, confirming the popup *target* was created fine -- the bug was specifically in mixing automatic and explicit attach for the same connection. Fixed by switching to `Target.setDiscoverTargets` (observe-only, CDP never auto-attaches anything itself) plus this client's own explicit `attachToTarget` call, reusing the exact same attach path `new_page` already used successfully.
2. **Chrome can transiently create and almost immediately destroy an extra, unrelated `page`-type target in the same browser context, around context/page setup.** With discovery-based attach working, `list_pages()` intermittently failed outright with a `Target.getTargetInfo` "No target with given id found" protocol error -- happening on the *first* call, before any popup had even been opened. A second, empty-URL `page` target (matching this session's own `browserContextId`) is discovered and successfully attached to alongside the real primary page, then disappears before its info can be queried. Root cause not fully isolated (plausibly an internal Chrome setup/teardown artifact of `Target.createBrowserContext`, not something this project's code requests), but the fix doesn't depend on knowing why: `list_pages()` now treats a `Target.getTargetInfo` failure for a page it just tried to query as "that page is already gone," drops it from the registry, and continues with the rest -- self-healing instead of surfacing a hard error for something the agent never asked about.
3. **A closing page's `Target.targetDestroyed` event isn't reliably delivered before `Target.getTargetInfo` on it starts failing.** The active-page-fallback logic (Decision #4) was originally implemented only in the `targetDestroyed`-event handler. Live testing (closing the popup via its own `window.close()` after switching to it) showed the fallback sometimes never ran: the destroy *event* lagged behind the target actually becoming unqueryable, so `list_pages()`'s bug-2 self-heal path removed the vanished page from the registry *without* resetting `active_target_id`, leaving it dangling and every page's `active` flag `false`. Fixed by factoring the "remove a page and fall back if it was active" logic into one `forget_page` helper used by *both* the event-driven path and the self-heal path, so either way of noticing a page is gone produces the same correct fallback.

Diagnosed via temporary `eprintln!` instrumentation directly in the affected code paths (removed after use) and repeated runs (5+) to distinguish a real race from a one-off flake, matching this project's established live-testing verification pattern.
