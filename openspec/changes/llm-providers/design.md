## Context

Four sequential changes turn `aib` into its own LLM agent harness: this one (provider client + config), OAuth/subscription auth, the agent loop, and MCP task delegation. Everything downstream depends on this change's types and config shape being right the first time, since `ChatRequest`/`ChatResponse`/`RoleClient` are the interface the agent loop (change 3) is built against -- so the design bar here is "won't need a breaking change later," not just "works for one client."

## Decision 1: Wire-neutral types, one enum client, not a trait

`Message`/`Part`/`ToolCall`/`ToolDef`/`ChatRequest`/`ChatResponse` in `types.rs` carry zero OpenAI-specific serde shape -- no `#[derive(Serialize)]`, no knowledge of `image_url`/`data:` URIs/the `tool_calls` wire array shape. Each provider's own client module (`client_compat.rs` for this change; a `client_responses.rs` change 2 adds) owns a private `Wire*` struct family and a `from_neutral`/`into_neutral` conversion pair. This means change 2's Responses-API shape (a genuinely different JSON structure -- `input` items, not `messages`; SSE events, not a single JSON body) is purely additive: a new module, a new `Client` enum variant, zero changes to `types.rs` or anything built on it.

`Client` is an enum (`Compat(CompatClient)`, with `Responses(ResponsesClient)` arriving in change 2), not `Box<dyn LlmClient>`. Two concrete backends, resolved once at config-load time, is a closed set -- a trait object would only buy dynamic dispatch this project never needs, at the cost of `async fn` in traits' current ergonomic friction (either `async-trait` boxing every call or `Box<dyn Future>` return types). `RoleClient { client, model, vision }` is the actual thing the agent loop holds; `vision` is read directly off it for change 3's screenshot-routing decision, no separate lookup needed.

## Decision 2: `CredentialSource` is a closed-but-growable enum behind one async method

Every client only ever calls `.bearer() -> Result<String>`. This change implements exactly one variant, `Static(String)` (a literal `api_key` or an `api_key_env` value read once at role-resolution time). Change 2 adds `OAuth { store, flow }`, whose `.bearer()` refreshes over the network when the stored token is near expiry -- and because `CompatClient`/`ResponsesClient` only ever call the same one async method, that addition touches zero call sites. The alternative (resolving a credential to a plain `String` once, up front) would have forced every OAuth-aware caller to re-resolve before each request instead of once at the trait boundary -- the async method is what makes "maybe refresh, maybe not" invisible to the client.

## Decision 3: `ProviderKind::OpenAiResponses` parses now, resolves to a clear error now

The config schema needs `kind = "openai-responses"` to be valid TOML from day one, or change 2 would need a breaking config-format change (users who wrote `openai-compat` configs against change 1 would need to touch anything else). So `ProviderKind` has both variants from the start; `Config::resolve_role` matches on kind and returns `LlmError::NotYetImplemented { kind: "openai-responses" }` for the unimplemented one -- honest about what doesn't work yet rather than silently miscompiling it into a `Compat` client that would send the wrong wire shape to `chatgpt.com`'s backend and fail confusingly instead of clearly.

## Decision 4: config precedence and "missing file is valid"

Resolution order: `--config <path>` (explicit, wins outright) → `AIB_CONFIG` env var (needed so tests and CI can point at a fixture config without touching the real one) → `./aib.toml` (project-local, lets a repo pin its own agent config) → `<data-dir>/aib/config.toml` (the per-user default, following the exact same `profile_base_dir()`-based convention every other `aib` subsystem already uses for profiles/traces/recordings). A file missing at whichever path this resolves to is `Config::empty()`, not an error -- `aib`'s browser-only tools (navigate/click/snapshot/...) must keep working with zero LLM configuration; only `aib llm ping`/the future agent commands fail, and they fail with a specific "role X isn't defined" or "provider Y has no credential" error rather than a generic "config missing."

## Decision 5: retry policy lives in the client, not the caller

`CompatClient::complete` retries HTTP 429 and 5xx up to twice more (3 attempts total) with exponential backoff (500ms, 1000ms), rebuilding the request fresh each attempt (not cloning a `reqwest::RequestBuilder` mid-flight, which the builder API doesn't cleanly support once `.json()` has consumed the body). A non-retryable 4xx (400, 401, ...) surfaces immediately as `LlmError::HttpStatus`. This lives in the client because every caller (the `aib llm ping` probe now, the agent loop in change 3) wants the same behavior -- provider hiccups shouldn't need per-call-site retry logic, and a caller that legitimately wants to abort fast can still do so by inspecting the returned error type.

## Bugs found during implementation

**`#[serde(rename_all = "kebab-case")]` on `ProviderKind` produced `open-ai-compat`, not `openai-compat`.** Caught by this module's own unit tests failing to parse a config that used the schema's documented `kind = "openai-compat"` string -- not assumed correct from reading the derive macro. Serde's case-conversion treats each capital letter as a new word boundary, so `OpenAiCompat` splits into `Open`/`Ai`/`Compat` → `open-ai-compat`, not the single-word "openai" every actual provider's docs and this config schema use. Fixed with explicit `#[serde(rename = "openai-compat")]`/`#[serde(rename = "openai-responses")]` per variant instead of relying on the automatic conversion -- a small, general lesson (recorded here rather than left implicit): `rename_all` is not a no-op for identifiers containing brand names/acronyms that don't follow standard word-boundary casing.

**`reqwest`'s workspace-level dependency entry has no TLS backend at all.** `Cargo.toml`'s existing `reqwest = { version = "0.13", default-features = false, features = ["json"] }` was added for `crates/mcp`'s in-process HTTP test client (local, plaintext `127.0.0.1`, no TLS ever needed there) -- discovered before writing any code, by checking `cargo info reqwest` rather than assuming a TLS feature name from an older reqwest version's docs (0.13 renamed the feature from `rustls-tls`, used in reqwest 0.11/0.12, to plain `rustls`). `crates/llm/Cargo.toml` adds `features = ["rustls"]` locally on top of the workspace-inherited dependency (cargo unions extra local features with the workspace's own), rather than changing the shared workspace entry and pulling TLS machinery into every other crate that happens to depend on `reqwest` for unrelated reasons.

## Testing note

Verified against a real local HTTP server (`crates/llm/tests/support/stub_server.rs`, a new hand-rolled body-reading/header-recording server in the style of `crates/engine/tests/support/http_server.rs`, which never needed to read a request body before this) for the real-socket round trip, Authorization header, and retry-on-5xx behavior -- not just serialized-JSON-shape unit tests, though those exist too (`client_compat.rs`'s own `#[cfg(test)]` module) for the cheap, fast structural checks. A live-provider smoke test (`tests/live_smoke.rs`) exists and is skipped, not failed, when no `DEEPSEEK_API_KEY` is set in the environment -- confirmed to skip cleanly here (no key available in this environment) and to actually work end-to-end via a separate manual check: `aib llm ping driver` against a real local Python HTTP stub (not part of the automated suite, a throwaway one-off), confirming the full CLI → config → role resolution → real HTTP call → output-formatting path works, not just the library layer in isolation.
