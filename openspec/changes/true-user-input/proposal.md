## Why

Every input mode built so far (instant, synthetic human-motion, trained human-motion) dispatches through CDP's `Input.dispatch*Event` — which, verified empirically during `human-motion-trained`, produces events Chrome itself marks `isTrusted: true`, but which are still ultimately *synthesized by the automation tooling*, not generated by the real OS input pipeline. PROPOSAL.md's Phase 4 ("True-user mode") exists for the applications that specifically try to detect *that* distinction — through OS-level signals CDP dispatch can't reach (real hardware input event metadata, driver-level timing characteristics some anti-bot systems fingerprint). This change adds that backend: real Windows `SendInput` calls, indistinguishable from a human's own mouse and keyboard at the OS level.

## What Changes

- A new, Windows-only OS input backend using `SendInput` (`Win32_UI_Input_KeyboardAndMouse`): moves the real system cursor and dispatches real keyboard events, not CDP-synthesized ones.
- Locates the browser's actual OS window (via `EnumWindows` filtered by the launched browser process's PID — CDP's own `Browser` domain doesn't expose a native window handle) and brings it to the foreground before dispatching, since `SendInput` targets whatever window currently has OS focus, unlike CDP dispatch, which targets a specific tab regardless of window focus.
- `browser_click`/`browser_type`'s existing `human_like`/`persona`/`trained_profile`/`seed` machinery is reused unchanged for *timing* (the same `mouse_path`/`typing_timeline` synthesis); only the final dispatch step changes from `Input.dispatchMouseEvent`/`dispatchKeyEvent` to `SendInput`, selected by a new `true_input: bool` parameter.
- Headed sessions only — `SendInput` targets a real, visible, focusable window; there's no real OS input surface for a headless renderer to receive it.

**Explicitly out of scope (follow-up changes), and why this is a meaningfully reduced first slice of the full Phase 4 vision:**
- **DPI/zoom calibration probe.** v1 assumes 100% display scaling and no browser zoom when converting CDP viewport coordinates to screen coordinates for `SendInput`. Getting this right across arbitrary DPI/zoom combinations is real, separate work (PROPOSAL.md's own exit criteria calls out 100/150/200% DPI and 80/100/125% zoom as a matrix to validate) -- v1 ships the mechanism and is honest that it's only calibrated for the common default case.
- **Multi-monitor / non-primary-display targeting.** `SendInput`'s absolute-coordinate mode normalizes against the primary display's metrics; v1 assumes the browser window is on the primary display.
- **A serialization queue for concurrent/interleaved real-input actions.** Not needed for this session's single-session-at-a-time model; a real concern only once multiple sessions could compete for OS input focus simultaneously.
- **Cross-platform support (macOS `CGEvent`, Linux `XTest`/`uinput`).** Windows-only for v1, matching the actual development/verification environment; other platforms return a clear "not supported on this OS" error rather than silently falling back to CDP dispatch.

## Capabilities

### New Capabilities
- `true-user-input`: real, OS-level (Windows `SendInput`) mouse/keyboard dispatch as an alternative to CDP-synthesized input, reusing the existing human-motion timing synthesis for *when* events fire while changing *how* they're actually delivered to the OS.

## Impact

- New dependency: `windows` (already transitively present in the dependency tree via other crates; adding it as a direct workspace dependency with the specific `Win32_UI_Input_KeyboardAndMouse`/`Win32_UI_WindowsAndMessaging`/`Win32_Foundation` features needed), Windows-only (`#[cfg(windows)]`), `cdp` crate.
- `crates/cdp/src/os_input.rs` (new, Windows-only): window discovery by PID, foreground-window activation, coordinate translation (viewport -> screen, 100% scale assumed), `SendInput`-based mouse move/click and keyboard dispatch.
- `crates/engine/src/session.rs`: `click_with`/`type_text_with` gain a `true_input: bool` flag on `HumanLike` (or an orthogonal parameter) selecting the OS dispatch path instead of CDP dispatch for the same synthesized timeline; rejected with a clear error for headless sessions or on non-Windows platforms.
- `crates/mcp/src/lib.rs`: `true_input: bool` parameter on `browser_click`/`browser_type`.
- Verification requires a real, visible headed browser window and will move the actual system mouse cursor and briefly take real keyboard focus -- explicitly confirmed with the user immediately before running, the same live-testing pattern used for `human-motion-trained`'s physical-human demo.
